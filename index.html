08 · Exhibit
▸ Decision · survival beats speculation

When the market moves against you faster than you can react

The most common failure mode in trading bots is the absence of a kill switch. A bot that keeps trading into a losing streak will eventually blow up the account. The fix isn't better entry signals — it's institutional-grade risk management. The Enhanced Risk Manager has 6 circuit breakers that halt trading when conditions deteriorate: daily loss limit (10%), consecutive losses, high volatility, API errors, total drawdown (20%), and manual halt. Each position has tiered profit-taking (close 50% at +10%, remainder at +20%), trailing stops, time-based exits, and maximum loss limits. The RiskStatus enum progresses from NORMAL to CAUTION to RESTRICTED to HALTED — giving the system graduated responses instead of binary on/off.

BEFORE: no risk management 
class SimpleBot:
  def place_bet(self, market, amount):
    # No position sizing
    # No stop loss
    # No circuit breakers
    # No portfolio heat check
    order = self.client.place_order(
      market=market,
      amount=amount  # Could be 100%
    )
    return order
  # If the market moves against us,
  # we keep betting. If the API fails,
  # we retry forever. If we hit 10
  # losses in a row, we bet #11.
  # This is how accounts get blown up.
AFTER: Enhanced Risk Manager 
class EnhancedRiskManager:
  def check_trade_allowed(self, size):
    # Circuit breaker 1: daily loss
    if self.daily_pnl <= -0.10 * bal:
      return HALTED, "Daily loss limit"
    # Circuit breaker 2: consecutive
    if self.consecutive_losses >= 3:
      return RESTRICTED, "Consecutive"
    # Circuit breaker 3: portfolio heat
    if self.portfolio_heat >= 0.50:
      return RESTRICTED, "Heat limit"
    # Circuit breaker 4: drawdown
    if self.total_drawdown >= 0.20:
      return HALTED, "Max drawdown"
    # Circuit breaker 5: API errors
    if self.api_errors >= 5:
      return HALTED, "API errors"
    # Circuit breaker 6: volatility
    if self.volatility_7d >= 2.0:
      return CAUTION, "High volatility"
    # Per-trade limit
    if size > 0.02 * bal:
      return RESTRICTED, "Oversized"
    return NORMAL, "OK"

  def check_exit(self, position):
    if position.pnl_pct >= 0.10:
      return TAKE_PROFIT_TIER1  # 50%
    if position.pnl_pct >= 0.20:
      return TAKE_PROFIT_TIER2  # rest
    if position.pnl_pct <= -0.05:
      return STOP_LOSS
    if position.hold_time >= 24:
      return TIME_EXIT
    return None  # Hold

10 · Decisions

Five questions worth answering slowly.

Short answer first — long answer if you need it.

Q01Why copy-trading instead of prediction?+

Predicting markets is hard. Identifying people who already predict well is easier. The bot copies proven edge instead of trying to generate it.

The efficient market hypothesis suggests that prediction markets are hard to beat. But not everyone in the market is equally skilled. The Winner Discovery Engine finds bettors with statistically significant edge and determines if copying their positions is +EV. This shifts the problem from "what will happen?" to "who knows what will happen?" — a fundamentally easier question.

Q02Why 1 bet per day?+

Over-trading is the silent killer of most bots. A hard limit forces patience, reduces transaction costs, and ensures every trade receives maximum analytical attention.

Traditional trading bots suffer from over-trading — they chase every signal, accumulate fees, and compound small losses into large drawdowns. A hard limit of 1 bet per day enforces discipline. It reduces API usage (10-20 calls/day vs hundreds). It ensures the EV Calculator, Multi-Factor Model, and Research Engine all run on every trade.

Q03Why CLI-only instead of a web dashboard?+

A web dashboard adds infrastructure, maintenance, and attack surface. A CLI is fast, lightweight, and fits the 15-minute daily workflow.

Web dashboards require a server, a database, authentication, SSL certificates, frontend frameworks, and ongoing maintenance. They introduce latency, security risks, and complexity. A CLI using Rich provides styled tables, progress bars, and syntax-highlighted reports in the terminal — fast, lightweight, and zero infrastructure.

Q04Why Python instead of a faster language?+

Python has the best ecosystem for data analysis (pandas, numpy, scipy) and the bot is API-bound, not CPU-bound. The bottleneck is network latency, not language speed.

The bot spends most of its time waiting for API responses and performing data analysis — tasks where Python excels. pandas and numpy are industry standards for quantitative analysis. scipy provides statistical functions that would require significant effort to reimplement in C++ or Rust. For the rare CPU-bound tasks, numpy's vectorized operations are implemented in C under the hood.

Q05Why Half-Kelly instead of full Kelly?+

Full Kelly is optimal for growth but extremely volatile. Half-Kelly reduces variance by 50% while retaining 75% of growth — a better tradeoff for real-world trading with estimation error.

The Kelly Criterion calculates the optimal bet size as (edge / odds). Full Kelly maximizes expected log wealth but produces extreme drawdowns. Half-Kelly reduces variance by 50% while retaining 75% of expected growth. This is the standard in professional trading because edge estimates are noisy and correlated bets reduce effective Kelly.

12 · Reflection

What surprised me. What I'd do differently. What's next.

Surprised by

How much the 1-bet-per-day limit improved backtest performance.

Simulations showed that bots with no trade limits accumulated 3x more fees and had 40% higher drawdowns — despite placing more trades. The constraint of patience turned out to be the highest-return feature. Also surprised by how effective the vanity gap detection was: approximately 15% of "winners" had inflated win rates from selective reporting, and filtering them out significantly improved copy performance.

Would do differently

Build the backtest engine before the real-time trading system.

v3.0.0 real-time features were exciting to build but hard to validate without historical simulation. A backtest-first approach would have caught that the predictive entry system had false-positive rates above acceptable thresholds in certain market conditions. The real-time components should have been validated against 6 months of historical data before implementation.

What's next

Live trading validation, ML retraining pipeline, institutional API.

Live trading validation with small capital ($100-500) for 3 months. Machine learning model retraining pipeline with weekly updates. Social features: leaderboards of bot users, anonymized performance sharing. Mobile push notifications for circuit breaker triggers. Institutional API for hedge fund integration. Regulatory compliance framework for multi-jurisdiction deployment.
